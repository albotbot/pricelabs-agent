    **YAML frontmatter** (must match monitoring skill pattern exactly):
    ```yaml
    ---
    name: pricelabs-analysis
    description: >

      detection with actionable recommendations, competitive market positioning,
      and demand calendar rendering.
    user-invocable: false

    ```

    **Intro paragraph:** "This skill provides analytical protocols for generating weekly optimization reports, detecting underperformance with specific recommended actions, analyzing competitive market positioning, and rendering demand calendars. Use these protocols in combination with pricelabs-monitoring and pricelabs-domain skills. All data access goes through MCP tools."

    **Section 1: Weekly Optimization Report Protocol** (ANLY-03, ANLY-04)
    Step-by-step execution flow when triggered by cron or user request:
    1. Compute portfolio KPIs: Call `pricelabs_get_portfolio_kpis` with `compare_to: "previous_week"` to get WoW data for all listings
    2. Compute STLY comparison: Call `pricelabs_get_portfolio_kpis` with `compare_to: "stly"` to get year-over-year data
    3. Detect underperformers: Call `pricelabs_detect_underperformers` with default thresholds to flag listings needing attention
    4. For each underperformer, compose a specific recommended action using the Underperformance Action Decision Tree (Section 2)
    5. Format the report using the Weekly Report Template (Section 5)
    6. Log the report: Call `pricelabs_log_action` with `action_type='report'` and description summarizing the report contents

    Include guidance on handling missing data:
    - "If WoW comparison returns null for prev_ fields: 'WoW comparison: not yet available (need 7+ days of daily snapshots. Collection began [earliest snapshot date].)'"
    - "If STLY comparison returns null: Use the revenue_vs_stly_pct field from listing_snapshots (PriceLabs API provides this natively). Only note unavailability if that field is also null."
    - "If no underperformers detected: State 'All listings performing within healthy thresholds' and skip the underperformance section."

    **Section 2: Underperformance Detection and Action Recommendations** (ANLY-01, ANLY-02)

    Threshold definitions with defaults:
    | Signal | Threshold | Severity |
    | Occupancy gap vs market | > 20% below market | Warning |
    | Revenue vs STLY | > 25% below STLY | Warning |
    | Health score decline | health_7_day AND health_30_day both < 50 | Warning |
    | Base price drift | > 15% from recommended_base_price | Info |
    | Price position: below 25th with high occ | Below p25, occ > market | Info |
    | Price position: above 90th with low occ | Above p90, occ < market - 10% | Warning |

    **Decision tree for recommended actions** (this is the ANLY-02 critical requirement -- actions MUST be specific with numbers, never vague):

    ```
    IF underperformance_type = 'occupancy_gap' OR 'occupancy_and_revenue':
      IF base_price > p75_price:
        ACTION: "Lower base price from $[current] to $[p50_price] (market median).
                 This will shift your price from the 75th+ percentile to the 50th,
                 which typically improves occupancy within 1-2 weeks."
      ELIF base_price > p50_price:
        ACTION: "Expand last-minute discount window. Consider extending your
                 discount curve from [current days] to [current + 7] days.
                 Your base price ($[current]) is near market median ($[p50_price]),
                 so the issue is likely last-minute availability, not base pricing."
      ELSE:
        ACTION: "Pricing is competitive (at or below median). Review listing
                 quality factors: photos, description, reviews, amenities.
                 Compare to top-performing comps in your neighborhood."

    IF underperformance_type = 'revenue_drop':
      IF occupancy is comparable to STLY (within 5%):
        ACTION: "ADR has dropped -- review base price. Current: $[base_price],
                 Recommended: $[recommended_base_price]. Consider increasing base
                 by $[diff] to restore ADR closer to STLY levels."
      ELIF occupancy significantly lower than STLY:
        ACTION: "Occupancy-driven revenue drop. Current occupancy: [X]%
                 vs STLY [Y]%. See occupancy gap recommendations above."
      ELSE:
        ACTION: "Check for cancelled reservations impacting revenue.
                 Review pricelabs_get_audit_log for recent cancellation alerts."

    IF underperformance_type = 'health_decline':
      ACTION: "Comprehensive listing audit recommended. Health scores are
               below 50 across both 7-day ([score]) and 30-day ([score]) windows.
               Check: photos freshness, description accuracy, amenity list
               completeness, review responses, competitive pricing position."
    ```

    **CRITICAL ANLY-02 rule:** Every recommendation MUST include:
    1. The specific metric that triggered the alert (e.g., "occupancy 25% below market")
    2. The specific action recommended (e.g., "lower base price from $185 to $165")
    3. The specific numbers involved (dollar amounts, percentages, percentile positions)
    Words like "consider", "may want to", or "look into" are NOT acceptable without specific numbers attached.

    **Section 3: Competitive Position Analysis Protocol** (ANLY-05)
    Step-by-step for when user asks "how is my pricing compared to the market?":
    1. Call `pricelabs_get_portfolio_kpis` to get market_position data (included in the KPI response)
    2. For each listing, present the positioning: "Listed Name is priced at $[base_price] -- positioned at [price_position] percentile. Market: p25=$[p25], p50=$[p50], p75=$[p75], p90=$[p90]. [N] comparable listings used."
    3. If listing is below 25th percentile with high occupancy: "You may be leaving revenue on the table. High occupancy at a low price point suggests room to increase base price."
    4. If listing is above 90th percentile with low occupancy: "Premium pricing may be limiting bookings. Consider moving toward the 75th percentile ($[p75_price]) to improve occupancy."
    5. Check market_snapshots freshness: if latest snapshot_date is older than 48 hours, call `pricelabs_get_neighborhood` for each listing first to refresh data, then call `pricelabs_store_market_snapshot` to persist it, before computing position.

    **Section 4: Demand Calendar Rendering Protocol** (ANLY-06)
    When user asks for a demand calendar:
    1. Call `pricelabs_get_prices` for the listing with the next 14 days (default range)
    2. For each date in the response, extract: date, price, demand_level (from computed fields), booking_status
    3. Render in this exact text format:
    ```
    Demand Calendar -- [Listing Name] (next [N] days)
    ---
    [Date] [Day]  $[Price]  [DEMAND]  [status]
    ...
    ---
    Legend: [HIGH] = Strong demand | [MED] = Moderate | [LOW] = Weak demand
    ```
    4. Demand level mapping (from computed-fields.ts demand_level): map demand_color values to HIGH/MED/LOW descriptors
    5. If user requests > 30 days: summarize instead of enumerating. Format: "Next [N] days: [X] high-demand dates, [Y] medium, [Z] low. Top high-demand dates: [list top 5-7]."
    6. If user didn't specify a range, default to 14 days. Offer: "Showing next 14 days. Want to see a longer range?"

    **Section 5: Report Templates**

    Weekly Optimization Report template:
    ```
    Weekly Optimization Report -- Week of [DATE]
    =============================================

    PORTFOLIO SUMMARY:
    Listings: [N] | Avg Occupancy: [X]% (WoW: [+/-Y]%) | Avg Revenue 7d: $[Z] (WoW: [+/-W]%)
    STLY Comparison: Revenue [+/-X]% | Occupancy [+/-Y]%

    UNDERPERFORMANCE ALERTS:
    [For each flagged listing:]
    [Warning icon] [Listing Name] -- [underperformance_type]
      Signal: [specific metric and value that triggered the flag]
      Recommended Action: [specific action from decision tree with dollar amounts]
      Market Context: Priced at $[base] (market p50: $[p50], position: [percentile])

    COMPETITIVE POSITIONING:
    [For each listing, one line:]
    [Name]: $[base] ([position] percentile) | Market p50: $[p50] | Occ: [X]% vs Market [Y]%

    WEEKLY TRENDS:
    [For each listing with WoW data:]
    [Name]: Occ [X]% -> [Y]% ([+/-Z]%) | Rev $[A] -> $[B] ([+/-C]%)

    ---
    Report generated [timestamp]. Data from [snapshot_date] snapshots.
    ```

    Underperformance Alert template (for immediate alerts outside weekly report):
    ```
    Underperformance Alert -- [Listing Name]
    Signal: [metric] is [value] ([threshold] threshold exceeded)
    Recommended Action: [specific action with numbers]
    Market Context: [competitive position]
    ```

    **Section 6: Coordination with Monitoring Skill**
    Clear delineation of responsibilities:
    - **Monitoring skill owns:** Daily health checks, real-time stale sync alerts, pace alerts (detection + basic context), cancellation alerts, interactive queries about current state
    - **Analysis skill owns:** Weekly optimization reports (deep analysis), underperformance detection with recommended actions, competitive position analysis, demand calendar rendering
    - **Alert dedup rule:** Before sending any underperformance alert, call `pricelabs_get_audit_log` filtered to `action_type='alert'` for that listing. If an alert for the same underperformance signal was sent within the last 24 hours, do NOT send a duplicate. This extends the existing 24h cooldown pattern from the monitoring skill.
    - **Cross-reference rule:** If a listing appears in both daily alerts this week AND the weekly report, the weekly report should note: "This listing was flagged [N] times in daily health checks this week for [signal]."
    - **Weekly report builds on daily data:** The weekly report uses snapshots already stored by the daily health check cron. It does NOT re-fetch from the PriceLabs API unless market data is stale (> 48 hours old). This prevents redundant API calls and rate limit consumption.
