**Create `mcp-servers/pricelabs/src/schemas/scale.ts`:**

Define 4 Zod schemas with `.describe()` on every field (established pattern):

1. `GetChangeImpactInputSchema` -- z.object with:
   - `listing_id`: z.string().optional() -- filter by listing ID
   - `pms`: z.string().optional() -- PMS name, required if listing_id provided
   - `pending_only`: z.boolean().optional() -- if true, only return changes with pending checks. Default false.
   - `limit`: z.number().optional() -- max entries. Default 20.

2. `RecordChangeInputSchema` -- z.object with:
   - `audit_log_id`: z.number() -- FK to audit_log.id for the execution entry
   - `listing_id`: z.string()
   - `pms`: z.string()
   - `change_type`: z.enum(["set_overrides", "update_listing", "delete_overrides"])
   - `affected_dates_start`: z.string().optional() -- for DSO changes (YYYY-MM-DD)
   - `affected_dates_end`: z.string().optional() -- for DSO changes (YYYY-MM-DD)
   - `before_json`: z.string() -- JSON snapshot of values before the change
   - `after_json`: z.string() -- JSON of what was changed to

3. `GetUserConfigInputSchema` -- z.object with:
   - `listing_id`: z.string().optional() -- get config for specific listing merged with globals
   - `pms`: z.string().optional() -- required if listing_id provided

4. `SetUserConfigInputSchema` -- z.object with:
   - `config_key`: z.enum(["occupancy_gap_threshold", "revenue_drop_threshold", "pace_lag_threshold", "health_score_threshold", "stale_sync_hours"])
   - `config_value`: z.number() -- numeric threshold value
   - `listing_id`: z.string().optional() -- set for specific listing, omit for global
   - `pms`: z.string().optional() -- required if listing_id provided

**Create `mcp-servers/pricelabs/src/tools/scale.ts`:**

Export `registerScaleTools(server: McpServer, db: Database.Database)`. This function receives only server and db (like Phase 2-3 tools, not the full service stack -- these tools do not need API access).

Register 4 tools:

1. **`pricelabs_record_change`** (writeHint tool):
   - Schema: RecordChangeInputSchema
   - Annotations: `readOnlyHint: false, destructiveHint: false, openWorldHint: false`
   - Handler: Compute check_7d_due, check_14d_due, check_30d_due by adding 7/14/30 days to current date. Call `insertTracking` from change-tracking queries. Return the created entry ID and the three due dates.
   - Date computation: Use `new Date()` for today, add days, format as YYYY-MM-DD.

2. **`pricelabs_get_change_impact`** (read-only tool):
   - Schema: GetChangeImpactInputSchema
   - Annotations: `readOnlyHint: true, destructiveHint: false, openWorldHint: false`
   - Handler: If `pending_only` is true, call `getPendingChecks` with today's date. Otherwise, if listing_id provided, call `getByListing`. If neither, call `getPendingChecks` with a far-future date to get all entries. Apply limit (default 20). Return the rows with parsed JSON fields for readability.

3. **`pricelabs_get_user_config`** (read-only tool):
   - Schema: GetUserConfigInputSchema
   - Annotations: `readOnlyHint: true, destructiveHint: false, openWorldHint: false`
   - Handler: If listing_id provided, call `getAllForListing` (merged per-listing + global). Otherwise call `getAllGlobal`. Also include the system defaults in the response so the agent always knows the effective values:
     ```typescript
     const SYSTEM_DEFAULTS = {
       occupancy_gap_threshold: 20,
       revenue_drop_threshold: -25,
       pace_lag_threshold: -20,
       health_score_threshold: 50,
       stale_sync_hours: 48,
     };
     ```
   - Return: `{ configs: [...rows], system_defaults: SYSTEM_DEFAULTS, scope: "global" | "listing" }`

4. **`pricelabs_set_user_config`** (write tool):
   - Schema: SetUserConfigInputSchema
   - Annotations: `readOnlyHint: false, destructiveHint: false, openWorldHint: false`
   - Handler: Validate config_value bounds:
     - `occupancy_gap_threshold`: must be >= 0 and <= 100
     - `revenue_drop_threshold`: must be <= 0 and >= -100 (it represents a decline)
     - `pace_lag_threshold`: must be <= 0 and >= -100
     - `health_score_threshold`: must be >= 0 and <= 100
     - `stale_sync_hours`: must be >= 1 and <= 720 (30 days max)
   - If listing_id provided but pms missing, return error. If listing_id missing and pms provided, return error (both or neither).
   - Call `upsertConfig` with listing_id as null if global scope. Return the saved config entry and whether it was an insert or update.

Follow the existing tool pattern: try/catch, JSON.stringify result in text content, isError: true on catch.

**Update `mcp-servers/pricelabs/src/tools/analysis.ts`:**

Modify the `pricelabs_detect_underperformers` handler to check user_config for thresholds before using hardcoded defaults. The tool's input params still take priority (for ad-hoc analysis with custom thresholds).

1. Add import: `import { createUserConfigQueries } from "../db/queries/user-config.js";`
2. In `registerAnalysisTools`, create the user config queries: `const userConfigQueries = createUserConfigQueries(db);`
3. In the detect_underperformers handler, replace the threshold defaulting logic:

Before (current):
```typescript
const occGapThreshold = params.occupancy_gap_threshold ?? 20;
const revenueStlyThreshold = params.revenue_stly_threshold ?? -25;
```

After (with user_config fallback):
```typescript
const occGapThreshold = params.occupancy_gap_threshold ?? (() => {
  const userValue = userConfigQueries.getConfigValue.get({
    config_key: 'occupancy_gap_threshold',
    listing_id: null,
    pms: null,
  });
  return userValue ? Number(userValue.config_value) : 20;
})();

const revenueStlyThreshold = params.revenue_stly_threshold ?? (() => {
  const userValue = userConfigQueries.getConfigValue.get({
    config_key: 'revenue_drop_threshold',
    listing_id: null,
    pms: null,
  });
  return userValue ? Number(userValue.config_value) : -25;
})();
```

Note: The user_config lookup here uses `listing_id: null, pms: null` to get the global threshold. Per-listing thresholds for detect_underperformers would require running the query per-listing, which is not how the current tool works (it queries all listings at once with a single threshold). For v1, global thresholds are sufficient for the batch query. Per-listing thresholds apply via the user_config tool directly when the agent evaluates individual listings.

4. Update the `thresholds` section of the response to indicate the source:
```typescript
thresholds: {
  occupancy_gap_pct: occGapThreshold,
  revenue_vs_stly_pct: revenueStlyThreshold,
  health_score: 50,
  source: params.occupancy_gap_threshold || params.revenue_stly_threshold ? 'parameter' : 'user_config_or_default',
},
```
