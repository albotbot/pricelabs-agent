**Section 8: Batch Approval Protocol (SCALE-01)**

Add after Section 7 (Rollback Protocol). This section formalizes the batch approval pattern that Section 6 already partially describes, adding error handling and batch reporting.

Content must include:

1. **Batch state tracking:** When presenting multiple recommendations from an optimization scan, number them clearly (already in Section 6 rule 3). After the user responds, the agent MUST echo back exactly which recommendations it will execute before proceeding. Example: "I will execute recommendations 1, 3, and 5. Recommendations 2 and 4 will be skipped. Confirm?"

2. **Batch approval syntax:** Recognize these patterns:
   - "approve all" -- execute all presented recommendations sequentially
   - "approve 1, 3, 5" or "approve 1 and 3" -- cherry-pick specific numbers
   - "reject all" or "skip all" -- reject all, log rejections
   - "approve 1-3, reject 4-5" -- mixed approval with ranges
   - Any ambiguous response -- echo back interpretation and ask for confirmation

3. **Sequential execution with error handling:** For each approved recommendation:
   - Execute the standard approval flow (Section 5): snapshot -> write -> verify -> log
   - If a write fails: log the failure (action_type='execution' with error details in details_json), report the error to the user, continue with the remaining approved recommendations. Do NOT abort the entire batch on a single failure.
   - Track results: keep a running tally of successes, failures, and skips

4. **Post-execution change tracking:** After each successful execution, call `pricelabs_record_change` with the audit_log_id from the execution log entry, the listing_id, pms, change_type, affected date range (for DSOs), before_json (from snapshot), and after_json (from write response). This registers the change for 7/14/30 day impact follow-up tracking.

5. **Batch completion report:** After all approved recommendations are processed, present a summary:
   ```
   Batch complete: Executed 3 of 5 recommendations.
   - Rec 1 (Mountain View Cabin orphan fill Mar 15): SUCCESS -- DSO -20% applied
   - Rec 2 (Beach House demand spike Mar 21-23): REJECTED by user
   - Rec 3 (Mountain View Cabin demand spike Apr 5-7): SUCCESS -- DSO +25% applied
   - Rec 4 (Beach House base price adjustment): REJECTED by user
   - Rec 5 (Lake House orphan fill Mar 18): FAILED -- rate limit exceeded, retry later

   Changes tracked for 7/14/30 day revenue impact follow-up.
   All changes sync to BookingSync during the next nightly cycle (6pm-6am CT).
   ```

6. **Batch audit logging:** After the batch is complete, log a single batch summary entry with `action_type='report'` containing the batch context: total recommendations, approved count, rejected count, failed count, and per-recommendation results in details_json.

**Section 9: Cancellation Fill Strategy Protocol (SCALE-03)**

Content must include:

1. **Trigger:** When new cancellations are detected during daily health checks (the `pricelabs_store_reservations` tool returns `new_cancellations` array) or when the user asks about recent cancellations.

2. **Step 1 -- Assess urgency.** For each newly cancelled reservation:
   - Calculate days until the freed check-in date
   - URGENT (< 7 days): Very short booking window. Any revenue is better than $0.
   - HIGH (7-14 days): Still bookable but time-sensitive. Moderate discounting.
   - MODERATE (14-30 days): Normal fill window. Mild discounting or no action if demand is healthy.
   - LOW (> 30 days): Algorithm and organic demand will likely handle it. Note for monitoring only.
   - Also assess: lost revenue amount (from cancelled reservation's rental_revenue), whether the freed dates create an orphan gap with adjacent bookings.

3. **Step 2 -- Check date availability.** CRITICAL: Before recommending a fill strategy, call `pricelabs_get_prices` for the freed dates and verify `unbookable != "1"` for each date. If dates are now owner-blocked, skip them. Also check existing DSOs -- do not layer fill discounts on top of existing DSOs without noting the interaction.

4. **Step 3 -- Formulate fill strategy by urgency:**
   - URGENT (< 7 days): Recommend aggressive DSO -25% to -30%. Present expected nightly price after discount.
   - HIGH (7-14 days): Recommend moderate DSO -15% to -20%.
   - MODERATE (14-30 days): Recommend mild DSO -10% to -15%, OR no action if demand_color shows healthy demand for those dates.
   - LOW (> 30 days): No DSO recommended. Note the cancellation for the next weekly report.

5. **Step 4 -- Present recommendation.** Format:
   ```
   Cancellation detected: [listing name]
   [guest info if available] cancelled [check-in] to [check-out] ($[revenue] lost).
   Urgency: [URGENT/HIGH/MODERATE/LOW] -- [X] days until check-in.
   Recommendation: [specific DSO with dates and percentage]
   Expected price: $[price] per night (down from $[current])
   ```

6. **Step 5 -- Track for follow-up.** If a fill strategy is approved and executed, call `pricelabs_record_change` to create a change_tracking entry. The 7-day follow-up will show whether the freed dates re-booked after the discount was applied.

7. **Integration with orphan day protocol.** If the freed dates from a cancellation create an orphan gap (1-3 night gap between remaining bookings), apply both the cancellation fill urgency assessment AND the orphan day protocol from Section 1. Use the more aggressive discount of the two.
