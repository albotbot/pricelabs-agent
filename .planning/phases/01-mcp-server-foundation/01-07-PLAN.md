**sync.ts -- registerSyncTools:**

Tool: pricelabs_push_prices
- Description: "Trigger a price sync/push for a listing. Forces PriceLabs to recalculate and push updated prices to connected OTAs. DESTRUCTIVE -- only use when prices need immediate refresh. REQUIRES reason."
- Input: { listing: z.string().describe("Single listing ID"), pms_name: z.string().describe("PMS identifier e.g. 'airbnb'"), reason: z.string().min(10) }
- Annotations: destructiveHint: true, readOnlyHint: false
- No cache (action, not data retrieval)
- Handler: Call apiClient.post("/v1/push_prices", { listing, pms_name }). No fetchWithFallback for writes. Return confirmation with reason.
- Note: PriceLabs API accepts a single listing per push call. If the agent needs to push multiple listings, it should call this tool once per listing.

Tool: pricelabs_add_listing
- Description: "Add a new listing to PriceLabs for price management. ONLY works for BookingSync PMS. DESTRUCTIVE -- creates a new managed listing. REQUIRES reason."
- Input: { listing_id: z.string(), pms_name: z.literal("bookingsync").describe("Only BookingSync PMS is supported for adding listings"), reason: z.string().min(10) } (plus any additional fields per API spec)
- Annotations: destructiveHint: true, readOnlyHint: false
- Handler: Call apiClient.post("/v1/add_listing_data", body). Invalidate listings cache. Return confirmation.
- Note: The PriceLabs API restricts add_listing_data to BookingSync PMS only. The z.literal("bookingsync") constraint enforces this at the schema level.

Both write tools: no fetchWithFallback, require reason, invalidate relevant cache.
