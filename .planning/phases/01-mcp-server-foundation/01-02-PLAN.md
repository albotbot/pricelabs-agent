Read `research/02-api-reference.md` for the full API specification. Create Zod schemas for each endpoint's input parameters and expected response shapes.

**common.ts** -- Shared schemas:
- `ListingIdSchema`: z.string() with .describe() for listing ID
- `PmsNameSchema`: z.string() with .describe() for PMS identifier (e.g., "airbnb", "vrbo")
- `DateStringSchema`: z.string().regex(/^\d{4}-\d{2}-\d{2}$/) for YYYY-MM-DD dates
- `CheckInOutSchema`: z.string().regex(/^[01]{7}$/) for 7-char binary Mon-Sun string
- `ToolResponseEnvelope`: Generic wrapper with `data`, `computed`, and `meta` (cache_age_seconds: number, data_source: "live"|"cached", api_calls_remaining: number, fetched_at: string). This enforces the locked decision that every response includes cache_age_seconds and data_source.

**listings.ts** -- Listing schemas:
- `GetListingsInputSchema`: { skip_hidden?: boolean, only_syncing?: boolean } -- input params for GET /v1/listings
- `GetListingInputSchema`: { listing_id: string, pms: string } -- input for GET /v1/listings/{id}
- `UpdateListingsInputSchema`: { listings: array of { id, pms, min?, base?, max?, tags? }, reason: string (min 10 chars) } -- locked decision: write tools require reason parameter
- Listing response schema matching API fields: id, pms, name, address, lat, lng, bedrooms, bathrooms, accommodates, currency, min_price, base_price, max_price, tags, health_7_day, health_30_day, health_60_day, occupancy_next_30, market_occupancy_next_30, revenue_past_7, stly_revenue_past_7, last_date_pushed, last_refreshed_at, recommended_base_price, etc.

**prices.ts** -- Price schemas:
- `GetPricesInputSchema`: { listing_id: string, pms: string, start_date: DateString, end_date: DateString, currency?: string }
- Price response schema: array of daily price entries with date, price, min_price, max_price, demand_color, demand_desc, booking_status, booking_status_STLY, ADR, ADR_STLY, unbookable

**overrides.ts** -- Override (DSO) schemas:
- `GetOverridesInputSchema`: { listing_id: string, pms: string, start_date?: DateString, end_date?: DateString }
- `SetOverridesInputSchema`: { listing_id: string, pms: string, overrides: array of { date, price_type ("fixed"|"percentage"), price_value (number), currency? (required if fixed), min_stay?, check_in?: CheckInOut, check_out?: CheckInOut }, reason: string (min 10 chars) }
- `DeleteOverridesInputSchema`: { listing_id: string, pms: string, dates: string[], reason: string (min 10 chars) }
- DSO validation: price_type "percentage" must have price_value in range -75 to 500. price_type "fixed" requires currency field.

**neighborhoods.ts:**
- `GetNeighborhoodInputSchema`: { listing_id: string, pms: string, start_date?: DateString, end_date?: DateString }
- Response: percentile data (25th, 50th, 75th, 90th) for ADR, occupancy, revenue

**reservations.ts:**
- `GetReservationsInputSchema`: { listing_id: string, pms: string, start_date: DateString, end_date: DateString } -- Note: start_date and end_date are REQUIRED by the PriceLabs API (not optional). The tool handler should provide sensible defaults if the agent doesn't specify (e.g., past 90 days to next 30 days), but the schema must not mark them as optional since the API will reject requests without them.
- Response: array of reservations with guest name, check_in, check_out, total, status, source, booked_at

Add `.describe()` to EVERY schema field -- these descriptions flow into MCP tool discovery and help the LLM understand what each parameter does.

All schemas must be exported as named exports. Do NOT define TypeScript interfaces separately -- types will be inferred in types.ts.

**types.ts** -- Infer all TypeScript types from Zod schemas using `z.infer<typeof Schema>`:

```typescript
import { z } from "zod";
import { GetListingsInputSchema, ... } from "./schemas/listings.js";
// ... import all schemas

export type GetListingsInput = z.infer<typeof GetListingsInputSchema>;
export type Listing = z.infer<typeof ListingResponseSchema>;
// ... all types
```

Also define the `ToolResponse<T>` generic type:
```typescript
export interface ToolResponse<T> {
  data: T;
  computed: Record<string, unknown>;
  meta: {
    cache_age_seconds: number;
    data_source: "live" | "cached";
    api_calls_remaining: number;
    fetched_at: string;
  };
}
```

**computed-fields.ts** -- Computed field calculators (locked decision: hybrid raw + computed):

`computeListingFields(listing: Listing): Record<string, unknown>`:
- `occupancy_gap_pct`: (market_occupancy_next_30 - occupancy_next_30) / market_occupancy_next_30 * 100. Returns null if either field missing or market is 0.
- `revenue_vs_stly_pct`: (revenue_past_7 - stly_revenue_past_7) / stly_revenue_past_7 * 100. Returns null if STLY is 0 or -1 (unavailable).
- `days_since_sync`: (Date.now() - Date.parse(last_date_pushed)) / 86400000. Returns null if last_date_pushed missing.
- `is_stale_sync`: days_since_sync > 2 (boolean). Threshold: 48 hours.
- `health_trend`: Compare health_7_day vs health_30_day to derive "improving", "stable", or "declining".

`computePriceFields(priceEntry: PriceEntry): Record<string, unknown>`:
- `demand_level`: Map demand_color hex to human label: "#FF0000"->"high", "#FFA500"->"medium-high", "#FFFF00"->"medium", "#00FF00"->"medium-low", "#0000FF"->"low". Fall back to demand_desc if present.
- `adr_vs_stly_pct`: (ADR - ADR_STLY) / ADR_STLY * 100. Returns null if either is -1.
- `is_booked`: Parse booking_status to determine if date is booked.

`computeNeighborhoodFields(neighborhood: NeighborhoodData, listing: Listing): Record<string, unknown>`:
- `price_percentile_position`: Where the listing's base_price falls relative to 25th/50th/75th/90th percentiles. Returns "below_25th", "25th_to_50th", "50th_to_75th", "75th_to_90th", "above_90th".

All functions must handle missing/null data gracefully -- return null for any computed field where source data is missing. Never throw on bad input data.

Export all compute functions as named exports.
