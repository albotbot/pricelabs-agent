  <action>
Create `scripts/validate-api.mjs` following the exact pattern of `scripts/validate-boot.mjs` (ESM, child_process spawn, stdio JSON-RPC, ANSI colors, check() helper, try/finally cleanup).

**Key differences from validate-boot.mjs:**
- Uses real PRICELABS_API_KEY from environment (NOT a test key)
- Fails fast with clear message if PRICELABS_API_KEY is not set: "Set PRICELABS_API_KEY to run API validation"
- Uses a temp database for isolation (same as boot validation)
- Budget: ~20 API calls total -- be conservative

**Script flow (in order):**

1. **Environment check** -- fail immediately if PRICELABS_API_KEY not set
2. **Build + start MCP server** -- same as validate-boot.mjs (npm run build, spawn with temp DB, initialize handshake)
3. **API Status (before)** -- call `pricelabs_get_api_status` (no args), record `rate_limit.remaining`. Print initial budget.
4. **LIVE-01: Get Listings** -- call `pricelabs_get_listings` (no required args). Parse the JSON response:
   - Check: response has `data` array with at least 1 listing
   - Check: first listing has `id` (non-empty string) and `pms` (non-empty string) and `name` (non-empty string)
   - Auto-discover: save `listing_id` and `pms` from first listing for subsequent calls
   - Print: "Discovered listing: {name} (id={listing_id}, pms={pms})"
   - Log the full first listing response as WARN-level diagnostic so user can see real field shapes

5. **LIVE-02: Get Prices** -- call `pricelabs_get_prices` with discovered listing_id, pms, start_date = today (YYYY-MM-DD), end_date = 30 days from today:
   - Check: response has `data` object with `data` array (daily price entries)
   - Check: at least 1 price entry exists
   - Check: first price entry has numeric `price` field > 0
   - Log any fields that are null/unexpected as WARN

6. **LIVE-03: Get Neighborhood** -- call `pricelabs_get_neighborhood` with discovered listing_id, pms:
   - Check: response has `data` object with nested `data` object
   - Check: "Listings Used" field exists (number or null)
   - Check: "Future Percentile Prices" exists with Y_values array
   - Log any unexpected shapes as WARN

7. **LIVE-04: Get Reservations** -- call `pricelabs_get_reservations` with discovered listing_id, pms, start_date = 90 days ago, end_date = 90 days from now:
   - Check: response has `data` object
   - Check: response has `data.data` array (reservation entries)
   - Note: array may be empty if no reservations -- that is OK, log it as INFO not FAIL
   - If non-empty: check first entry has `check_in`, `check_out` fields

8. **LIVE-05: Cache verification** -- call `pricelabs_get_listings` again (same args as step 4):
   - Measure wall-clock time for both the first call (step 4) and this second call
   - Check: second call's response includes `meta.data_source: "cached"` or `meta.cache_age_seconds > 0`
   - Print timing comparison: "First call: {X}ms, Second call (cached): {Y}ms"

9. **LIVE-05: Rate limit tracking** -- call `pricelabs_get_api_status` again:
   - Check: `rate_limit.remaining` is less than the initial value from step 3 (API calls were consumed)
   - Print: "Rate limit: {initial} -> {current} ({consumed} calls consumed)"

10. **Summary** -- print pass/fail count, same format as validate-boot.mjs

**Critical implementation details:**
- JSON-RPC message IDs: start at 1 for initialize, use incrementing IDs for each tools/call
- Parse tool response: `response.result.content[0].text` is a JSON string that must be JSON.parsed to get the tool's data
- The tool's data is a `ToolResponse<T>` envelope: `{ data, computed, meta }` -- the actual API data is inside `.data`
- Timing: wrap the sendJsonRpc + waitForResponse in Date.now() pairs for each call
- On WARN (unexpected field shape): log it clearly but do NOT increment failures -- per user decision, mismatches are WARN not FAIL
- waitForResponse timeout should be 30000ms (real API calls take longer than boot validation)
- Use date calculation: `new Date().toISOString().split('T')[0]` for today, similar for +30d/-90d
- Error in tool response: check `response.result.isError === true` -- if so, log the error text and FAIL that check

Also create `scripts/validate-api.sh` shell wrapper (same pattern as validate-boot.sh):
```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."
node scripts/validate-api.mjs "$@"
```
