  <name>Task 1: Create automated messaging validation script</name>
  <files>scripts/validate-messaging.mjs, scripts/validate-messaging.sh</files>
  <action>
Create `scripts/validate-messaging.mjs` following the established validation script pattern from Phase 6-9 (validate-boot.mjs, validate-deployment.mjs, etc.). Use the same check/pass/fail/summary reporting format.

The script reads JSON and markdown files from disk (no network calls, no Docker, no MCP server spawn needed). It validates:

**Section 1: Channel Configuration (openclaw.json)**
- Slack channel config exists and has `enabled: true`
- Slack has `socketMode: true`, `threadReply: true` (required for reply-based approvals per CONTEXT.md)
- Slack has `appToken` and `botToken` env var references (${SLACK_APP_TOKEN}, ${SLACK_BOT_TOKEN})
- Telegram channel config exists and has `enabled: true`
- Telegram has `botToken` env var reference (${TELEGRAM_BOT_TOKEN})
- Telegram groups have `requireMention: true`
- Agent session dmScope is `per-channel-peer` (for proper DM scoping)

**Section 2: Cron Job Targeting**
- 4 cron jobs exist in jobs.json (2 daily, 2 weekly)
- Daily Slack job: schedule `0 8 * * *`, tz `America/Chicago`, channel `slack`, staggerMs 0
- Daily Telegram job: schedule `0 8 * * *`, tz `America/Chicago`, channel `telegram`, staggerMs 30000
- Weekly Slack job: schedule `0 10 * * 1`, tz `America/Chicago`, channel `slack`, staggerMs 0
- Weekly Telegram job: schedule `0 10 * * 1`, tz `America/Chicago`, channel `telegram`, staggerMs 30000
- All jobs have `sessionTarget: "isolated"` (prevent cross-job interference)
- All jobs have `delivery.bestEffort: true` (graceful failure on delivery issues)

**Section 3: Protocol Name Matching**
This is the critical cross-reference check. Extract protocol names referenced in each cron job's payload.message, then verify each one appears as an exact `## Protocol Name` heading in the corresponding skill file.

Protocols referenced by daily cron:
- "Daily Health Check Protocol" -> must exist in monitoring-protocols.md
- "Cancellation Fill Strategy Protocol" -> must exist in optimization-playbook.md
- "Revenue Impact Assessment" -> check monitoring-protocols.md (referenced via the revenue impact instruction)

Protocols referenced by weekly cron:
- "Weekly Optimization Report Protocol" -> must exist in analysis-playbook.md
- "Orphan Day Detection Protocol" -> must exist in optimization-playbook.md
- "Demand Spike Detection Protocol" -> must exist in optimization-playbook.md
- "Base Price Calibration Check" -> must exist in optimization-playbook.md
- "Recommendation Prioritization" -> must exist in optimization-playbook.md

For each protocol reference, use regex to find the `## {Protocol Name}` heading in the skill file content. Report PASS/FAIL for each match.

**Section 4: Health Summary Format Validation**
Parse monitoring-protocols.md and verify the Daily Health Check Protocol contains:
- Portfolio count line (`Portfolio: {n} listings active`)
- Sync status line (`Sync Status:`)
- Occupancy line (`Avg Occupancy`)
- Revenue line (`Revenue (past 7d)`)
- Three severity tiers (Critical Alerts, Warnings, Opportunities)

This validates MSG-01/MSG-04 content format requirements.

**Section 5: Approval Flow Configuration**
Verify the infrastructure supports reply-based approvals (MSG-03):
- Slack `threadReply: true` in openclaw.json (replies go to thread, not channel)
- optimization-playbook.md contains recommendation output format
- domain-knowledge.md references write safety gate
- PRICELABS_WRITES_ENABLED is set to "false" in openclaw.json MCP server env (safety default)

**Section 6: Environment Variable Inventory**
Collect all ${VAR} references across openclaw.json and jobs.json. Output a summary table of required env vars for messaging:
- SLACK_APP_TOKEN (source: openclaw.json channels.slack)
- SLACK_BOT_TOKEN (source: openclaw.json channels.slack)
- TELEGRAM_BOT_TOKEN (source: openclaw.json channels.telegram)
- SLACK_HEALTH_CHANNEL (source: jobs.json daily/weekly slack delivery.to)
- TELEGRAM_HEALTH_CHAT_ID (source: jobs.json daily/weekly telegram delivery.to)

Report count of unique env vars and which files reference them.
