**1. domain-knowledge.md** (~80-120 lines)
- PriceLabs platform overview: what it is, how dynamic pricing works, the HLP algorithm
- Key terminology: base price, min/max price, DSO (date-specific override), demand color, health scores, STLY, ADR, RevPAR, occupancy, booking pace
- API capabilities summary: which MCP tools are available, what each category does (read tools, store tools, write tools, analysis tools)
- Portfolio management context: typical portfolio sizes, rate budgets (1000/hr), cache/rate limiter behavior
- Write safety: PRICELABS_WRITES_ENABLED gate, never enable without explicit user permission

**2. monitoring-protocols.md** (~100-150 lines)
- **Daily Health Check Protocol** (must match cron reference exactly):
  - Step 1: Fetch all listings via `pricelabs_get_listings`
  - Step 2: Store snapshots via `pricelabs_store_daily_snapshots`
  - Step 3: Compute booking pace -- compare occupancy_next_30 vs market_occupancy_next_30
  - Step 4: Detect stale syncs -- flag listings where days_since_sync > 2 (48 hours)
  - Step 5: Revenue comparison -- compare revenue_past_7 vs stly_revenue_past_7, flag if < 70%
  - Step 6: Generate health summary with Critical/Warning/Info tiers
  - Step 7: Store reservations via `pricelabs_store_reservations` to detect cancellations
- Alert threshold table: stale sync (>48h = Critical), low occupancy (<80% of market = Warning), revenue drop (>30% vs STLY = Warning), health score issues (Warning)
- Stale sync handling: what to report, when to trigger sync
- Revenue impact assessment protocol: check pending assessments via `pricelabs_get_change_impact`

**3. analysis-playbook.md** (~80-120 lines)
- **Weekly Optimization Report Protocol** (must match cron reference exactly):
  - Step 1: Fetch all listings and current snapshots
  - Step 2: Compute portfolio KPIs: ADR, RevPAR, Occupancy, Avg LOS, Avg Lead Time
  - Step 3: Calculate week-over-week deltas from stored snapshots
  - Step 4: Calculate STLY comparisons using reservation data
  - Step 5: Identify underperforming listings (occupancy < 80% of market, revenue < 70% of STLY)
  - Step 6: Generate specific recommended actions per listing
- KPI calculation formulas: ADR = total revenue / booked nights, RevPAR = total revenue / available nights, occupancy = booked nights / available nights
- Underperformance detection criteria and thresholds
- Report format template matching the output format from agent/workflows.md

**4. optimization-playbook.md** (~100-160 lines)
- **Orphan Day Detection Protocol**:
  - Scan next 30 days for 1-3 night unbookable gaps between bookings
  - Exclude owner blocks (unbookable dates)
  - For each orphan gap: listing name, dates, gap length, recommended DSO discount (20% for 1-2 nights, 15-20% for 3-4 nights)
  - Present fill strategies with specific date ranges
- **Demand Spike Detection Protocol**:
  - Scan next 90 days for clusters of high-demand dates (demand_color = red)
  - Check if existing DSOs already cover those dates
  - Check if algorithm has already elevated pricing
  - Recommend event-based percentage overrides (+15% to +50% based on demand intensity)
- **Base Price Calibration Check**:
  - Compare listing base_price to neighborhood percentiles (p25, p50, p75) via `pricelabs_get_neighborhood`
  - Compare to recommended_base_price
  - Flag if deviation > 10% and last review > 30 days
  - Show full min/max price impact for recommended changes
- **Recommendation Prioritization**:
  - Priority order: orphan days first (immediate revenue), demand spikes second (upcoming events), base price third (long-term)
  - Present top 3-5 highest-priority recommendations requiring approval
  - Each recommendation: listing, action type, specific values, expected impact
- **Cancellation Fill Strategy Protocol** (Section 9):
  - Assess urgency based on check-in proximity: <7 days = URGENT, 7-14 days = HIGH, 14-30 days = MEDIUM, >30 days = LOW
  - For URGENT/HIGH: verify date availability, check current pricing, recommend last-minute discount adjustment or DSO
  - For MEDIUM: monitor rebooking pace, recommend DSO if pace is slow after 7 days
  - For LOW: no immediate action, monitor in next daily check
