  <name>Task 1: Create Dockerfile for OpenClaw sandbox simulation</name>
  <files>Dockerfile</files>
  <action>
Create a `Dockerfile` at the project root that mimics OpenClaw's Docker sandbox environment. Per user decision, use a multi-stage build for efficiency.

**Stage 1: Build**
```dockerfile
FROM node:20-slim AS builder
WORKDIR /app
# Copy package files for the MCP server
COPY mcp-servers/pricelabs/package*.json mcp-servers/pricelabs/
# Install dependencies
RUN cd mcp-servers/pricelabs && npm ci --production=false
# Copy source and build
COPY mcp-servers/pricelabs/ mcp-servers/pricelabs/
RUN cd mcp-servers/pricelabs && npm run build
```

**Stage 2: Runtime**
```dockerfile
FROM node:20-slim
# Install SQLite (needed for better-sqlite3 native module runtime)
RUN apt-get update && apt-get install -y --no-install-recommends sqlite3 && rm -rf /var/lib/apt/lists/*
WORKDIR /app
# Copy built MCP server with node_modules
COPY --from=builder /app/mcp-servers/pricelabs/dist/ mcp-servers/pricelabs/dist/
COPY --from=builder /app/mcp-servers/pricelabs/node_modules/ mcp-servers/pricelabs/node_modules/
COPY --from=builder /app/mcp-servers/pricelabs/package.json mcp-servers/pricelabs/package.json
# Copy OpenClaw config and skills
COPY openclaw/ openclaw/
# Copy agent docs (read-only reference)
COPY agent/ agent/
# Environment variables (defaults -- overridden at runtime)
ENV PRICELABS_API_KEY=""
ENV PRICELABS_DB_PATH="/data/pricelabs.sqlite"
ENV PRICELABS_BASE_URL="https://api.pricelabs.co"
ENV PRICELABS_WRITES_ENABLED="false"
# Create data directory for SQLite
RUN mkdir -p /data
# The MCP server communicates via stdio JSON-RPC, not HTTP
# OpenClaw spawns it as a subprocess, so no EXPOSE needed
CMD ["node", "mcp-servers/pricelabs/dist/index.js"]
```

IMPORTANT: better-sqlite3 is a native Node.js addon. The build stage and runtime stage must use the same Node.js major version and OS architecture so the native binary is compatible. Using node:20-slim for both stages ensures this.

Do NOT copy .env files, secrets, or node_modules from the host. The Dockerfile must be self-contained (npm ci in build stage).

Do NOT include PRICELABS_API_KEY in the image -- it is injected at runtime via docker run -e.

  <name>Task 2: Create deployment validation script</name>
  <files>scripts/validate-deployment.mjs, scripts/validate-deployment.sh</files>
  <action>
Create `scripts/validate-deployment.mjs` following the EXACT established pattern from `scripts/validate-boot.mjs`:
- Same ESM (.mjs) structure with top-level await
- Same ANSI color helpers (PASS/FAIL), check() function, failure counter
- Same try/finally cleanup pattern

Per user decision: Docker validation should work WITHOUT PRICELABS_API_KEY (testing infrastructure, not API connectivity).

The script validates all 5 DEPLOY requirements:

**DEPLOY-01: MCP server runs inside Docker container**
1. Run `docker build -t pricelabs-agent-test .` from the project root. Check exit code 0.
2. Start a container in the background with a test API key:
   `docker run -d --name pricelabs-deploy-test -e PRICELABS_API_KEY=test-deploy-key -e PRICELABS_DB_PATH=/data/test.sqlite pricelabs-agent-test`
   Wait 2-3 seconds for the server to start.
3. Verify the container is running: `docker inspect --format='{{.State.Running}}' pricelabs-deploy-test`
4. Verify the MCP server process is running inside the container:
   `docker exec pricelabs-deploy-test pgrep -f "node.*dist/index.js"` or check process list.
5. Test MCP server responds to tool/list inside the container by using docker exec to send a JSON-RPC request via stdin:
   `echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"deploy-test","version":"1.0.0"}}}' | docker exec -i pricelabs-deploy-test node mcp-servers/pricelabs/dist/index.js`

   Actually, since the CMD is already running the server, we need a different approach. Instead:
   - Stop the existing container
   - Run a new container interactively that pipes JSON-RPC via stdin:
   ```
   const proc = spawn('docker', ['run', '--rm', '-i',
     '-e', 'PRICELABS_API_KEY=test-deploy-key',
     '-e', 'PRICELABS_DB_PATH=/data/test.sqlite',
     'pricelabs-agent-test'
   ], { stdio: ['pipe', 'pipe', 'pipe'] });
   ```
   Then use the same sendJsonRpc/waitForResponse pattern from validate-boot.mjs to send initialize and tools/list requests.
6. Verify tools/list returns 28 tools (same check as validate-boot.mjs BOOT-03).

**DEPLOY-02: Environment variables injected correctly**
1. During the docker run from DEPLOY-01, the server started with test API key and custom DB path.
2. Verify by checking that the server initialized successfully (it reads env vars on startup).
3. Additionally, run: `docker exec pricelabs-deploy-test printenv PRICELABS_API_KEY` should return the test key.
   Actually, since we're using the interactive spawn approach: verify by checking that initialize response is successful (server reads PRICELABS_API_KEY on startup -- if missing or wrong format, it would fail differently). The tool/list success proves env vars were injected.
4. Verify DB path: After the server runs, check that `/data/test.sqlite` exists inside the container:
   Run a separate `docker run --rm pricelabs-agent-test ls -la /data/` or check after the interactive session.

**DEPLOY-03: Skill files loaded and referenced**
This can be validated WITHOUT Docker (file system checks):
1. Check all 4 skill files exist: `openclaw/skills/domain-knowledge.md`, `openclaw/skills/monitoring-protocols.md`, `openclaw/skills/analysis-playbook.md`, `openclaw/skills/optimization-playbook.md`
2. Check each file is non-empty (> 50 lines each).
3. Parse `openclaw/openclaw.json` and verify the agent entry has `instructions` array containing all 4 skill file paths.
4. Verify skill files are also present inside the Docker image:
   `docker run --rm pricelabs-agent-test ls openclaw/skills/` should list all 4 files.

**DEPLOY-04: Daily health check cron jobs valid**
1. Read and parse `openclaw/cron/jobs.json` as JSON.
2. Find the 2 daily health check jobs: `daily-portfolio-health-slack` and `daily-portfolio-health-telegram`.
3. Verify each has:
   - `schedule.kind === "cron"`
   - `schedule.expr === "0 8 * * *"` (8am daily)
   - `schedule.tz === "America/Chicago"`
   - Telegram job has `schedule.staggerMs === 30000` (30-second stagger)
   - `payload.kind === "agentTurn"`
   - `payload.message` contains "Daily Health Check Protocol"
   - `delivery.channel` matches (slack/telegram respectively)

**DEPLOY-05: Weekly optimization cron jobs valid**
1. Find the 2 weekly optimization jobs: `weekly-optimization-report-slack` and `weekly-optimization-report-telegram`.
2. Verify each has:
   - `schedule.expr === "0 10 * * 1"` (Monday 10am)
   - `schedule.tz === "America/Chicago"`
   - Telegram job has `schedule.staggerMs === 30000`
   - `payload.message` contains "Weekly Optimization Report" and "Orphan Day Detection"
   - `delivery.channel` matches (slack/telegram respectively)
