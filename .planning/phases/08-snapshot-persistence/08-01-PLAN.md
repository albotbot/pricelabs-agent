The script validates all 5 requirements in sequence:

**STORE-01: Store Daily Snapshots**
1. Call `pricelabs_get_listings` to fetch real listing data (reuse pattern from validate-api.mjs)
2. Transform listings into the format expected by StoreDailySnapshotsInputSchema:
   - Each listing needs: listing_id (from `id` field), pms, name, health scores, occupancy data, revenue data, base_price, recommended_base_price, last_date_pushed, computed fields (occupancy_gap_pct, revenue_vs_stly_pct, days_since_sync, is_stale_sync), data_json (full JSON stringified listing)
   - Handle the fact that API response fields may be strings ("43 %") or numbers -- the store tool normalizes via toNum()
3. Call `pricelabs_store_daily_snapshots` with the transformed snapshots array
4. Call `pricelabs_get_snapshots` with `table: "listing_snapshots"` to read back data
5. Verify: check response has `stored` count matching listing count, and get_snapshots returns rows with matching listing_id values

**STORE-02: Store Price Snapshots**
1. Pick the first listing from STORE-01 results (use its listing_id and pms)
2. Call `pricelabs_get_prices` with that listing to fetch real price data (reuse pattern from validate-api.mjs -- POST body format: `{listings: [{id, pms, dateFrom, dateTo}]}` with dateFrom=today, dateTo=today+30d)
3. Transform prices into format expected by StorePriceSnapshotsInputSchema:
   - listing_id, pms at top level
   - prices array: each entry needs price_date (from `date` field), price (from `price` field), demand_level (from `demand_level` or `color`), booking_status, booking_status_stly, adr, adr_stly
4. Call `pricelabs_store_price_snapshots` with the listing_id, pms, and prices array
5. Call `pricelabs_get_snapshots` with `table: "price_snapshots"`, listing_id, pms to read back
6. Verify: stored count matches prices count, and get_snapshots returns rows

**STORE-03: Store Reservations + Cancellation Detection**
1. Call `pricelabs_get_reservations` for the same listing (reuse validate-api.mjs pattern)
2. Transform reservations into StoreReservationsInputSchema format:
   - listing_id, pms at top level
   - reservations array: reservation_id (from `id` or reservation identifier), check_in, check_out, booked_date, booking_status, rental_revenue, total_cost, no_of_days, currency, data_json (stringified full reservation)
3. Call `pricelabs_store_reservations` with the full batch -- this is the first pass (real data)
4. Verify: upserted count matches reservation count
5. **Cancellation simulation (second pass):**
   - Take the reservations array and remove the first reservation (simulating it disappeared from the API)
   - Modify the removed reservation to have `booking_status: "cancelled"` and store it separately
   - Call `pricelabs_store_reservations` again with the modified reservation (booking_status changed to "cancelled")
   - The upsert logic in reservations.ts CASE expression should detect the status change and set `cancelled_on = datetime('now')`
6. Query back: Call `pricelabs_get_snapshots` with `table: "listing_snapshots"` (note: reservations are not in get_snapshots -- use a direct SQL approach). Actually, the MCP server does NOT have a tool to query reservations directly from the get_snapshots tool (only listing_snapshots, price_snapshots, market_snapshots). Instead, verify via the `new_cancellations` field in the store_reservations response -- the second pass call should return `new_cancellations` with the simulated cancellation.
7. Verify: new_cancellations array in the response contains the cancelled reservation_id with cancelled_on populated

**STORE-04: Store Market Snapshot**
1. Call `pricelabs_get_neighborhood` for the same listing (reuse validate-api.mjs pattern)
2. Transform neighborhood data into StoreMarketSnapshotInputSchema format:
   - snapshots array with: listing_id, pms, listings_used, p25_price, p50_price, p75_price, p90_price, market_occupancy, data_json (full stringified response)
   - Extract percentile prices from the neighborhood response -- per Phase 7, Future Percentile Prices uses Category/Labels keys. Look for price percentile data in the response shape.
3. Call `pricelabs_store_market_snapshot`
4. Call `pricelabs_get_snapshots` with `table: "market_snapshots"`, listing_id, pms
5. Verify: stored count is 1, and get_snapshots returns rows

**SAFE-02: Pre-Write Snapshot**
1. Call `pricelabs_snapshot_before_write` with the same listing_id, pms, operation_type: "set_overrides", start_date: today, end_date: today+30d, channel: "test"
2. Verify response contains: snapshot_type, listing_id, pms, captured_at, listing_state (with base_price), existing_overrides
3. Verify audit_log has entry: use the MCP tool `pricelabs_get_audit_log` if it exists, or use `pricelabs_get_snapshots` -- actually, the audit log is not queryable via get_snapshots. Check if there is a `pricelabs_get_audit_log` tool. Looking at the tool registration, Phase 2 created `pricelabs_log_action` and `pricelabs_get_audit_log`. Use `pricelabs_get_audit_log` with action_type: "snapshot" to verify the audit entry exists.
4. Verify: audit_log response contains at least 1 entry with action_type "snapshot" and details_json that is valid JSON containing listing_state

Important implementation notes:
- The real API data shapes were validated in Phase 7. Use the same field access patterns discovered there.
- The `pricelabs_get_listings` response returns listings with `id` (not `listing_id`) and `pms` fields.
- The `pricelabs_get_prices` requires POST body `{listings: [{id, pms, dateFrom, dateTo}]}` -- use the corrected format from Phase 7.
- The `pricelabs_get_neighborhood` response has Future Percentile Prices with Category/Labels keys.
- Cancellation detection relies on the SQL CASE expression in reservations.ts upsert -- changing booking_status to 'cancelled' on a previously 'booked' reservation triggers `cancelled_on = datetime('now')`.
- The snapshot_before_write tool makes live API calls (fetches listing and overrides). It requires PRICELABS_API_KEY.
- Budget approximately 8-10 API calls: get_listings (1), get_prices (1), get_reservations (1), get_neighborhood (1), snapshot_before_write listing fetch (1), snapshot_before_write overrides fetch (1), plus possible cache hits. Well within 1000/hr budget.
- If `pricelabs_get_audit_log` does not exist as a registered tool, instead verify SAFE-02 by checking the snapshot_before_write response itself (it returns the snapshot payload directly). The audit_log storage is an internal side effect that the tool performs, so the returned snapshot data proves it was captured.
