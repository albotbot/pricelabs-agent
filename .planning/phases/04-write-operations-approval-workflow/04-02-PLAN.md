**Section 1: Orphan Day Detection Protocol (OPT-05)**

Steps the agent follows:
1. For each listing, call `pricelabs_get_prices` for the next 30 days.
2. Scan `booking_status` for each date. Map to: booked, available, blocked.
3. CRITICAL: Check the `unbookable` field for every date. If `unbookable = "1"`, that date is an owner block -- exclude it from gap analysis entirely.
4. Identify orphan gaps: consecutive available dates surrounded by booked or blocked dates.
   - 1-night gap: highest priority (single night between bookings, most likely to go unbooked)
   - 2-night gap: high priority (short stay, may need min-stay reduction)
   - 3-night gap: moderate priority (may book naturally, monitor first)
5. For each orphan gap found:
   - Check current `min_stay` for those dates. If min_stay exceeds the gap length, that is the cause of the orphan (the gap cannot be booked because min_stay is too high).
   - Calculate recommended fill price: current price * 0.80 (20% orphan discount).
   - For 1-night gaps caused by min_stay: recommend reducing min_stay to 1 via DSO override.
   - For 2-3 night gaps: recommend a percentage DSO of -15% to -20%.
6. Never recommend changes for dates that are already booked (`booking_status = "booked"`).
7. Present all orphan gaps grouped by listing. For each recommendation include:
   - Listing name and dates affected
   - Current price and proposed price (or current min_stay and proposed min_stay)
   - Why this is an orphan (what bookings surround it)
   - Expected revenue impact: "Filling a 1-night gap at $X earns $X vs $0 if it stays empty"

**Section 2: Demand Spike Detection Protocol (OPT-06)**

Steps:
1. For each listing, call `pricelabs_get_prices` for the next 90 days.
2. Identify demand spikes: dates where `demand_color` indicates high demand. Map colors:
   - `#FF0000` or similar red shades = HIGH demand
   - `#FFA500` or orange = MEDIUM-HIGH demand
   - Group consecutive high-demand dates into event clusters.
3. For each demand spike cluster:
   - Call `pricelabs_get_overrides` to check if DSOs already exist for those dates.
   - If DSOs already exist, check whether the current algorithm price is already elevated significantly above base. If so, note that the algorithm is already handling this event and skip the recommendation unless there is specific reason to believe the algorithm is under-pricing.
   - If no DSOs exist and the algorithm has not significantly elevated pricing: recommend a percentage DSO between +15% (moderate demand) and +30% (extreme demand, multi-day events).
   - Use percentage DSOs (not fixed-price) for demand spikes. Percentage DSOs work with the algorithm rather than overriding it entirely.
4. Cross-reference with `pricelabs_get_neighborhood` for market validation. If neighborhood occupancy for the period is also elevated, this confirms the demand signal.
5. Never recommend DSOs for already-booked dates.
6. Present each recommendation with:
   - Listing name and dates affected
   - Demand level (HIGH/MEDIUM-HIGH) and source (demand_color)
   - Current algorithm price vs proposed DSO-adjusted price
   - Whether existing DSOs are in place
   - Market context from neighborhood data

**Section 3: Monthly Base Price Calibration Protocol (OPT-07)**

Steps:
1. Before recommending any base price change, check the audit log for the last base price change for this listing: call `pricelabs_get_audit_log` with `listing_id` and `action_type='execution'`. Scan results for base price changes. If the last change was less than 30 days ago, skip this listing. The 30-day minimum interval between base price changes is a hard rule to prevent panic pricing.
2. Call `pricelabs_get_neighborhood` for the listing to get market percentile data (p25, p50, p75, p90 prices and market occupancy).
3. Call `pricelabs_get_listing` for the current base price, min price, max price, and the PriceLabs `recommended_base_price`.
4. Compare base price to market percentiles:
   - Below p25 with occupancy above market average: listing is under-priced and in high demand. Recommend increase toward p50.
   - Above p75 with occupancy below market average: listing is over-priced and struggling. Recommend decrease toward p50.
   - Between p25 and p75 with normal metrics: no change needed.
5. Cross-reference with PriceLabs `recommended_base_price`. If it differs significantly from current base (more than 10%), factor this into the recommendation.
6. CRITICAL: Show the full cascading impact of a base price change. PriceLabs min and max prices can be configured as percentages of base price. When presenting a base price change recommendation, calculate and show:
   - Current base, min, max prices
   - Proposed base, and resulting new min and max (if they are percentage-based)
   - "Current base: $185, min: $111 (60%), max: $555 (300%). Proposed base: $210. New min: $126, new max: $630."
7. Only recommend one direction of change at a time. Never recommend two consecutive base price decreases for the same listing without an intervening review period.
8. Present the recommendation with:
   - Current base vs proposed base
   - Market percentile positioning (current and proposed)
   - Listing occupancy vs market occupancy
   - PriceLabs recommended_base_price for reference
   - Full min/max price impact

**Section 4: Write Operation Safety Protocol**

These rules apply to ALL write operations (set_overrides, update_listings, delete_overrides):

1. **Pre-write snapshot is MANDATORY.** Before EVERY write operation, you MUST call `pricelabs_snapshot_before_write` with the listing_id, pms, operation_type, and date range (if applicable). NEVER call `pricelabs_set_overrides`, `pricelabs_update_listings`, or `pricelabs_delete_overrides` without first calling `pricelabs_snapshot_before_write`. No exceptions.

2. **Never write without approval.** Do NOT execute any write operation without explicit user approval. "Sounds good", "sure", "ok" are NOT approval. The user must say "approve" or "yes, proceed" or an unambiguous equivalent. If uncertain, ask again: "Just to confirm -- shall I proceed with this change? Reply 'approve' to execute."

3. **Post-write verification.** The `pricelabs_set_overrides` tool already performs post-write verification and reports any silently dropped dates. After receiving the write response, check the `verification` field:
   - `"verified"`: All dates confirmed. Report success.
   - `"partial"`: Some dates dropped. Report which dates were dropped and why (usually past dates or outside sync window).
   - `"unverified"`: Verification fetch failed. Note this and suggest the user check manually.

4. **Sync timing caveat.** After every successful write, inform the user: "Changes saved to PriceLabs. They will sync to [PMS name] during the next nightly sync cycle (6pm-6am CT). For immediate sync, I can trigger a manual push with push_prices." Only offer push_prices if the user indicates urgency.

5. **One write at a time.** Execute write operations sequentially, not in parallel. Wait for one write to complete and verify before starting the next. This prevents cascading failures if a write is rejected by PriceLabs.

**Section 5: Approval Flow Protocol (OPT-01, OPT-02)**

When presenting a pricing recommendation that requires approval:

**Step 1: Pre-write snapshot.** Call `pricelabs_snapshot_before_write` to capture current state. This snapshot was likely just captured during the detection protocol, but if more than 30 minutes have passed since the recommendation was generated, re-capture a fresh snapshot immediately before execution.

**Step 2: Present the recommendation.** Format EVERY recommendation with ALL of these fields:
- **What will change:** Specific field (base price / DSO percentage / DSO fixed price / min-stay)
- **Current value:** From the snapshot (exact numbers)
- **Proposed new value:** The recommendation (exact numbers)
- **Rationale:** Which detection protocol triggered this (orphan day / demand spike / base price calibration) and the specific metrics that support it
- **Impact:** For base price: show min/max cascade. For DSOs: show which dates affected and resulting prices. For min-stay: show which dates and new minimum.
- **Listing name and ID** for clarity

**Step 3: Log the recommendation.** Call `pricelabs_log_action` with:
- `action_type: 'recommendation'`
- `listing_id` and `pms`
- `description`: One-line summary (e.g., "Recommend 20% orphan discount for Mountain View Cabin, March 15")
- `details_json`: JSON with `{ change_type, listing_id, pms, current_values, proposed_values, rationale, detection_protocol }`

**Step 4: Wait for approval.** Present: "Reply **approve** to proceed or **reject** to skip this change."
Do NOT proceed without explicit approval. Do NOT interpret ambiguous responses as approval.

**Step 5: On approval:**
1. Log approval: `pricelabs_log_action` with `action_type: 'approval'`, details including `approved_by: 'user'` and `approved_at` timestamp.
2. Re-snapshot if more than 30 minutes since last snapshot (Pitfall 6 from research -- stale snapshots).
3. Execute the write: call the appropriate write tool (`pricelabs_set_overrides` for DSOs, `pricelabs_update_listings` for base/min/max changes).
4. Log execution: `pricelabs_log_action` with `action_type: 'execution'`, details including full before/after values from snapshot and write response, verification status, and any dropped dates. Use this structure for `details_json`:
   ```json
   {
     "change_type": "set_overrides",
     "listing_id": "...",
     "pms": "...",
     "approved_by": "user",
     "approved_at": "ISO timestamp",
     "before": { "base_price": ..., "overrides": [...] },
     "after": { "overrides": [...] },
     "rationale": "...",
     "verification": "verified|partial|unverified",
     "dropped_dates": []
   }
   ```
5. Confirm to user with before/after values and sync timing caveat.

**Step 6: On rejection:**
1. Log: `pricelabs_log_action` with `action_type: 'approval'`, details noting rejection and user's reason if provided.
2. Acknowledge: "Got it, skipping this change."
3. Move to the next recommendation if any.

**Section 6: Recommendation Prioritization**

When running an optimization scan (triggered by cron or user request), prioritize recommendations to prevent approval fatigue:

**Priority levels:**
- **HIGH (present immediately):** Orphan days within the next 14 days. These represent imminent, certain revenue loss. Gap will remain empty if not filled.
- **MEDIUM (present after high-priority):** Demand spikes with no DSOs in the next 30 days. Revenue upside from pricing into confirmed demand.
- **LOW (present only if asked):** Base price calibration. Monthly check, not time-sensitive. Only recommend if 30+ days since last adjustment.

**Presentation rules:**
1. Present a maximum of 5 recommendations per scan. If more than 5 opportunities detected, present the top 5 by priority (HIGH first, then MEDIUM) and say: "I found [N] more opportunities. Want to see the rest?"
2. Group recommendations by listing when possible. Present all recommendations for one listing together rather than jumping between listings.
3. When presenting multiple recommendations, number them clearly: "Recommendation 1 of 3:", "Recommendation 2 of 3:", etc.
4. After presenting, the user can approve individually ("approve 1"), approve all ("approve all"), or reject all ("reject all" or "skip").
5. Recommendations older than 48 hours should be recalculated rather than executed, as prices and availability may have changed.

**Section 7: Rollback Protocol**

When the user requests a rollback of a previous change:

1. **Find the snapshot.** Call `pricelabs_get_audit_log` with `listing_id` and `action_type='snapshot'` to find the pre-write snapshot for the change to be rolled back.
2. **Show what will be restored.** Present the before values from the snapshot and the current values (which are the post-change values).
3. **Re-fetch current state.** CRITICAL: Before executing rollback, call `pricelabs_snapshot_before_write` again to capture the CURRENT state. This prevents overwriting any manual changes the user may have made between the original write and the rollback request.
4. **Get approval.** Rollback is still a write operation. Present: "I will restore the following values: [before values]. Reply 'approve' to proceed."
5. **Execute rollback.** Use the appropriate write tool to re-write the before values:
   - For DSO rollback: call `pricelabs_set_overrides` with the original override values, or `pricelabs_delete_overrides` if the original state had no overrides for those dates.
   - For base price rollback: call `pricelabs_update_listings` with the original base/min/max values.
6. **Log the rollback.** Call `pricelabs_log_action` with `action_type: 'rollback'`, including the snapshot reference, before/after values, and reason for rollback.
7. **Verify.** Post-write verification applies to rollbacks too. Confirm the values were restored correctly.
