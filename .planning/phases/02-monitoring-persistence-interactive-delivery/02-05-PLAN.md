    ---
    name: pricelabs-monitoring
    description: >
      Portfolio monitoring and reporting protocols. Teaches the agent how to
      perform daily health checks, generate formatted reports, track booking
      pace, detect stale syncs, and answer interactive portfolio queries.

    **Opening paragraph:**
    This skill provides the operational protocols for monitoring portfolio health and answering user questions. Use these protocols in combination with pricelabs-domain skill's analytical framework. All data access goes through MCP tools -- never rely on conversation context for current data.

    **Section 1: Daily Health Check Protocol** (MON-01, MON-02, MON-05)
    When triggered for a daily health check (by cron or user request):
    1. Call `pricelabs_get_api_status` to check rate limit budget before batch operations
    2. Call `pricelabs_get_listings` to fetch all active listings with health scores and computed fields
    3. For each listing, note:
       - Health trend (from computed health_trend field)
       - Occupancy gap vs market (from computed occupancy_gap_pct)
       - Revenue vs STLY (from computed revenue_vs_stly_pct)
       - Sync freshness (from computed is_stale_sync and days_since_sync)
    4. Call `pricelabs_store_daily_snapshots` with all listing data to persist today's snapshot
    5. For each listing, call `pricelabs_get_prices` to fetch next 90 days of prices, then call `pricelabs_store_price_snapshots` to persist
    6. For each listing, call `pricelabs_get_neighborhood` and then `pricelabs_store_market_snapshot` to persist market data
    7. Call `pricelabs_get_reservations` for each listing, then `pricelabs_store_reservations` to persist and detect cancellations
    8. Call `pricelabs_get_booking_pace` for each listing to get pace at 7/30/60/90 day cutoffs vs STLY
    9. Identify alerts:
       - Stale syncs: any listing where is_stale_sync = true (>48 hours). Before alerting, call `pricelabs_get_audit_log` filtered to action_type='alert' for that listing to check if an unresolved stale sync alert was sent in the last 24 hours. Only alert again after 24h cooldown or if the sync recovered and went stale again.
       - Pace behind STLY: any listing where is_behind_stly = true (>20% behind) at any cutoff window
       - New cancellations: reported by pricelabs_store_reservations
    10. Call `pricelabs_log_action` with action_type='report' to log the daily health check execution
    11. Format and deliver the summary report (see Report Formatting section)

    **Section 2: Report Formatting** (MON-02, DEL-01)
    Format the daily portfolio health summary in this structure. Use plain text with structure -- OpenClaw converts to channel-native formats.

    ```
    Portfolio Health Summary -- [DATE]

    ALERTS (if any):
    [alert emoji] [listing name]: [alert description with specific numbers]

    PORTFOLIO OVERVIEW:
    Listings: [total] | Synced: [synced/total] | Stale: [stale count]
    Avg Occupancy: [X]% (Market: [Y]%) | Gap: [Z]%
    Revenue vs STLY: [+/-X]%

    PER-LISTING DETAIL:
    [For each listing, one compact block:]
    [listing name]
    Health: [7d/30d/60d] [trend arrow] | Occ: [X]% (Mkt: [Y]%, Gap: [Z]%)
    Revenue 7d: $[X] vs STLY $[Y] ([+/-Z]%)
    Pace 30d: [X] bookings / $[Y] vs STLY [A] bookings / $[B] ([+/-C]%)
    Base: $[X] (Rec: $[Y]) | Last Sync: [relative time]
    ```

    Use up/down arrows for trends (improving/declining/stable). Keep each listing to 4-5 lines max. For alerts, lead with the most urgent (stale syncs first, then pace alerts, then cancellations).

    **Section 3: Booking Pace Tracking** (MON-03, MON-04)
    Booking pace measures forward bookings compared to STLY at the same point in time. The cutoffs:
    - 7-day: Immediate booking momentum (volatile, use for trend detection)
    - 30-day: Short-term pace (primary alert trigger)
    - 60-day: Medium-term pace (confirms sustained trends)
    - 90-day: Seasonal pace (strategic indicator)

    Alert threshold: 20% behind STLY at 30-day or longer cutoffs. Do NOT alert on 7-day alone (too volatile).
    When alerting, include: which cutoff triggered, specific numbers (current vs STLY), and the percentage gap. Suggest reviewing the listing's pricing strategy using the pricelabs-domain skill's optimization playbook.

    **Section 4: Stale Sync Detection** (MON-05)
    A stale sync is >48 hours since `last_date_pushed`. This means PriceLabs prices are not reaching the PMS/channel manager.
    Severity: HIGH -- stale syncs mean the listing is showing incorrect prices.
    When detected:
    1. Check if alert was already sent in last 24 hours (dedup via audit log)
    2. If new alert needed: describe the issue, how long the sync has been stale, and suggest checking PMS connection status
    3. Log the alert via `pricelabs_log_action` with action_type='alert'

    **Section 5: Interactive Query Protocol** (INT-01 through INT-04)
    When a user asks about portfolio performance:
    1. ALWAYS fetch live data first -- call `pricelabs_get_listings` (never rely on conversation history or stale context)
    2. Include data freshness in every response (from meta.cache_age_seconds and meta.data_source)
    3. For listing-by-name queries (INT-02): match the user's input against listing name, city_name, state, and tags. If ambiguous, list the top matches and ask for clarification.
    4. For comparative queries (INT-03): fetch all listings, rank by the requested metric, present top N with numbers
    5. For trend questions: call `pricelabs_get_snapshots` to retrieve historical data, compare current to past
    6. For pace questions: call `pricelabs_get_booking_pace` with the relevant listing
    7. For "what changed?" questions: call `pricelabs_get_audit_log` to show recent actions

    Always state the data source and freshness: "Based on live data fetched just now..." or "Based on cached data from 15 minutes ago..."

    **Section 6: Approval Flow** (DEL-03)
    When presenting a recommendation that requires approval:
    1. State the recommendation clearly with specific numbers
    2. Explain the rationale (why this change, what data supports it)
    3. Show what will change (before -> after values)
    4. Ask: "Reply 'approve' to proceed or 'reject' to skip."
    5. Wait for the user's response
    6. On approval: execute the action, log via pricelabs_log_action with action_type='approval'
    7. On rejection: acknowledge, log via pricelabs_log_action with action_type='approval' (with rejection in details)

    Do NOT make any pricing changes without explicit user approval. This is the most critical safety rule.

    **Create `openclaw/cron/jobs.json`** with two cron jobs for dual-channel delivery (as recommended by research -- approach A, two separate jobs):

    ```json
    [
      {
        "name": "daily-portfolio-health-slack",
        "schedule": {
          "kind": "cron",
          "expr": "0 8 * * *",
          "tz": "America/Chicago",
          "staggerMs": 0
        },
        "sessionTarget": "isolated",
        "wakeMode": "next-heartbeat",
        "payload": {
          "kind": "agentTurn",
          "message": "Run the daily portfolio health check. Follow the Daily Health Check Protocol from your monitoring skill. Fetch all listings, store snapshots, compute booking pace, detect stale syncs, and generate the portfolio health summary report.",
          "model": "opus",
          "thinking": "high"
        },
        "delivery": {
          "mode": "announce",
          "channel": "slack",
          "to": "${SLACK_HEALTH_CHANNEL}",
          "bestEffort": true
        }
      },
      {
        "name": "daily-portfolio-health-telegram",
        "schedule": {
          "kind": "cron",
          "expr": "0 8 * * *",
          "tz": "America/Chicago",
          "staggerMs": 30000
        },
        "sessionTarget": "isolated",
        "wakeMode": "next-heartbeat",
        "payload": {
          "kind": "agentTurn",
          "message": "Run the daily portfolio health check. Follow the Daily Health Check Protocol from your monitoring skill. Fetch all listings, store snapshots, compute booking pace, detect stale syncs, and generate the portfolio health summary report.",
          "model": "opus",
          "thinking": "high"
        },
        "delivery": {
          "mode": "announce",
          "channel": "telegram",
          "to": "${TELEGRAM_HEALTH_CHAT_ID}",
          "bestEffort": true
        }
      }
    ]
    ```

    Note: 30-second stagger on Telegram job to avoid concurrent API bursts from two parallel agent runs.

    The delivery channel/chat IDs use environment variable references (`${SLACK_HEALTH_CHANNEL}`, `${TELEGRAM_HEALTH_CHAT_ID}`) that the user configures in `~/.openclaw/.env`.
